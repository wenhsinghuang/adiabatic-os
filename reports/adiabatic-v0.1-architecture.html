Adiabatic OS 是什麼、現在怎麼跑

這份 report 把目前實作整理成一張圖：SQLite/D0 是 source of truth， pages/ 是 working tree mirror，apps/ 是可 vibe-code 的獨立 app， 所有資料 mutation 經過 Guard。

一句話模型

使用者累積的是 append-only event log；app、pages、D2 tables、shell UI 都只是這個 log 上方的操作面或 derived/read model。

資料主權

workspace 在本機；OS update 和 app code update 不應該動到使用者資料。

原生 primitives

保留 SQL、DDL、Git、filesystem、React/Node，不用自訂 app DSL。

安全邊界

不信任 app；write path 由 runtime/bridge/Guard 注入身份、檢查 grants、寫 D0 audit。

Workspace Layout

workspace 是使用者資料所在的 local vault-like directory，但它不是 Git repo 的必要邊界。

workspace/
├── CLAUDE.md
├── pages/
│   └── *.mdx
├── apps/
│   └── <app-id>/
│       ├── .git/
│       ├── manifest.json
│       ├── package.json
│       └── index.tsx
└── runtime-managed data

邊界

  • pages/：D1 working tree mirror，coding agent 可直接改。
  • apps/<id>/：每個 app 自己一個 Git repo，管理 code/update/rollback。
  • data 不進 app Git；schema/data mutation 走 Guard。
  • AI-facing docs 不主動說 DB 位置，也不把 DB 當操作面。

D0 / D1 / D2

D0
Raw event log
append-only、source injected、schema_version required。
真正 source of truth。未來 derive 邏輯變了，可以 replay/rederive。
D1
Pages working tree
pages/*.mdx mirror 到 DB docs table。
File -> DB source 是 working-tree:pages。D1 write 存 git-style patch；delete 存 full snapshot。
D2
Derived/read-model tables
app 透過 raw SQL DML 寫入。
每次 INSERT/UPDATE/DELETE 產生 CDC-style D0 payload：before/after/primary_key/sql/params。

Write Path：所有 mutation 都過 Guard

Caller
shell、working tree listener、CLI、WebContainer app。
→
Runtime identity
host/bridge 注入 source，app 不能自己傳。
→
Guard
檢查 DML/DDL allowlist、system table protection、manifest write grants。
→
SQLite + D0 audit
mutation 和 audit 在同一個 transaction 內完成。
API允許什麼D0 audit
writeEvent(event) 直接寫 D0 event；source 由 Guard 注入。 event 本身就是 D0。
writeDoc(id, content) upsert D1 doc。 d1.write，payload 內含 git-style patch。
deleteDoc(id) 刪 D1 doc。 d1.delete，payload 內含 full snapshot。
write(sql, params) 單一 INSERT / UPDATE / DELETE，不可碰 system tables。 d2.insert / d2.update / d2.delete，CDC-style payload。

Schema Lifecycle：promote / demote

Promote = 增加結構

  • CREATE TABLE
  • CREATE INDEX
  • ALTER TABLE ... ADD COLUMN

Demote = 減少結構

  • DROP TABLE
  • DROP INDEX
CLI / app request
adiabatic promote "<ddl>" 或 API request。
→
Shell approval
default 需要 modal approval；可記住 trusted coding agent。
→
Guard transaction
dry-run、system table protection、schema snapshot、DDL、D0 audit。

這裡保留 raw DDL，不做 ops DSL。Guard 只限制「DDL 類型」和「不能碰 system tables」。

App Runtime / Sandbox

App code
apps/<id>/index.tsx，普通 React/Node project。
→
WebContainer
bundle app、serve /apps/<id>/、host bridge server。
→
Shell tab
載入 WebContainer app URL。app browser page CSP：runtime network default deny。
→
@adiabatic/system
只 call /system/*；bridge 從 referer 推 app id，再轉給 core。
問題目前答案
app 能不能直接傳 X-App-Id？ 不能。bridge 不信任 app header，從 app URL / referer 推導 identity，再給 core X-Adiabatic-App-Id。
app 能不能任意 network？ browser runtime 透過 CSP 限制 connect-src 'self'。完整 connector/ETL network policy deferred。
coding agent 開發是不是自由？ 是。它仍然寫普通 React/TS/Node；安全性在 runtime boundary + Guard，不在 app DSL。

CLI 給 Coding Agent 的最小操作面

adiabatic query "SELECT * FROM events ORDER BY created_at DESC LIMIT 20"
adiabatic promote "CREATE TABLE focus_sessions (id TEXT PRIMARY KEY, started_at INTEGER)"
adiabatic demote "DROP TABLE scratch"

沒有 D1 write CLI；D1 最自然的操作方式是直接改 pages/*.mdx。 D2 row writes 主要由 app runtime 的 system.write() 做。

Update / Versioning Boundary

OS update

  • local patch archive：npm run package:patch
  • 更新 shell/core/Guard/runtime。
  • workspace path 記在 Electron userData，不因 OS update 重設。

App update

  • 每個 app 自己 .git。
  • Git 管 code/update/rollback。
  • data/migration 不進 Git，走 Guard promote/demote。
