Q: 有沒有可能 先做某些事 未來能讓substrate改動

有,而且這個問題其實打到了 substrate 設計的核心難題——**怎麼在不夠完備的時候 ship,但保留未來改動的權利**。

這不是個新問題。長壽的 substrate 都解過,而且解法有 pattern 可循。讓我把這些 pattern 整理出來,然後看哪些對你的處境適用。

---

## 一、先講一個關鍵觀念:不是所有改動都同樣難

「substrate 改動」這四個字其實混了好幾種非常不同的東西,難度差幾個數量級:

**Tier 1:加東西(加 attribute、加 source、加 derive function)**

- 幾乎不會傷到舊使用者
- 新東西舊版本看不到,但舊資料還能用
- 任何半像樣的設計都能支援這個

**Tier 2:改 derive 邏輯(同樣的 raw event,衍生方式變了)**

- 如果你保留了 raw event,這完全可以——重新 derive 就好
- 如果你只保留了 derived 結果,做不到
- **這層是 substrate 設計最重要的分水嶺**

**Tier 3:改 event schema 本身(欄位改名、語義改變、新增必要欄位)**

- 需要 migration
- 可以做,但要設計
- 有 pattern 可以讓這變便宜

**Tier 4:改 storage 物理結構(從 sqlite 換成 jsonl、加密 boundary 改變)**

- 需要 dump + reload
- 痛但可以做,只要 dump 格式是穩的

**Tier 5:改 ontology 本身(從 event-based 換成別的)**

- 不可能不傷使用者
- 等同於做新產品

你的真正擔心應該是 Tier 2、3、4,因為 Tier 1 自動可行、Tier 5 不該想。下面講的所有 pattern 都是針對 Tier 2-4。

---

## 二、第一個關鍵 pattern:Raw 跟 Derived 必須分層,而且 raw 永遠贏

這是 Hickey 也好、event sourcing 也好、其實所有可演化系統的共通祕密:

**只要你保留了完整的 raw event log,大部分改動都變便宜,因為你可以用新的邏輯 re-derive 整個歷史。**

具體怎麼做:

1. **Raw event 是 append-only 而且 schemaless 到極致**。第一版的 raw event 不應該預設任何 derive 怎麼用它。它只記錄發生了什麼,不假設這個資訊之後會被怎麼解讀。
2. **所有 derived state(D1、D2、aggregate、pattern、trait)都標記為 derived**,而且帶 `derived_from_event_ids` 跟 `derived_by_version`。任何時候你都可以丟掉所有 derived 重新算。
3. **Derive 邏輯是程式碼,raw event 是資料**。程式碼可以隨便改,資料必須穩定。

這個分層的副作用:**你可以在 substrate v0.1 的時候只 ship raw event store,而把所有 derive 邏輯先放鬆**。Derive 邏輯第一年是 throwaway 的,你愛怎麼改怎麼改,使用者只要相信「原始的東西在,結論可以重算」就不會生氣。

對你的具體 implication:**第一個版本最值得做對的是 raw event schema,其他都可以後補**。如果你只能做對一件事,就做對這個。其他層你愛怎麼亂搞都可以,只要 raw 沒污染。

---

## 三、第二個關鍵 pattern:Schema 演化的標準技巧

假設第一年你 ship 了一個 raw event schema,第二年你發現某個欄位設計錯了。怎麼辦?有幾個成熟 pattern:

**Pattern A:版本標籤 + 多版本共存**

每個 event 帶一個 `schema_version` 欄位。新 event 用新版,舊 event 還是舊版,讀取時根據版本走不同 parser。Postgres、Kafka、protobuf 都這樣做。

優點:零 migration,完全向後相容。 缺點:程式碼複雜度線性增加(支援的版本越多,讀取邏輯越多)。

**Pattern B:Lazy migration**

舊 event 不動,但讀取時 transform 成新 schema。Migration 邏輯寫在讀取層,不是寫入層。

優點:存在的歷史不被改寫(維持 immutability),但 derived 結果用統一 schema。 缺點:每次讀都要付 transform 成本,雖然小。

**Pattern C:Migration event**

寫一個特殊 event 說「從這個時間點開始 schema 版本變了」,然後新舊 event 並存。Derive 時自動處理跨版本的情況。

優點:歷史保留、版本變遷本身也是 fact。 缺點:derive 邏輯要懂得處理版本邊界。

**Pattern D:雙寫期**

新版本上線後,一段時間內同時用新舊 schema 寫 event(同一個事件寫兩份)。等夠長之後丟掉舊版讀取邏輯。

優點:轉換期可以 rollback。 缺點:儲存成本翻倍,要設好雙寫期長度。

實務上,大型 event-sourced 系統通常**混用 A 跟 B**:重要的 schema 變動用版本標籤共存,小的(欄位改名、預設值)用 lazy migration。

對你的 implication:第一版的 event schema 必須**強制帶 version 欄位**,即使第一版只有一個版本。這個欄位是你未來改動權的入場券,沒有它你就被鎖死。

## 四、第三個關鍵 pattern:把所有可能變的東西放在資料裡而不是程式碼裡

這是 Datomic 學派的核心信念,也是最反直覺的設計決定:

**如果某個東西未來可能變,把它寫成資料,不要寫成程式碼或固定 schema。**

例子:

- **不要把 source 種類寫成 enum**(`SLACK | EMAIL | SCREEN | MEETING`),寫成 string。未來加新 source 不需要改 schema,加個新 string 而已。
- **不要把 actor 寫成固定的 user/system 二元**,寫成一個有結構的 reference,可以擴展成 user / system / agent / external_party / unknown。
- **不要把 attribute set 寫死**,讓 event payload 是個 open-ended map,使用者(或你之後)可以加任意欄位。
- **不要把 promotion criteria 寫進 substrate 程式碼**,把它寫成 rule 或 config,使用者可以替換。

這個原則的代價是 type safety 變弱、靜態檢查變難、初期可能會有不一致的資料。但它換來的是**幾乎所有「我當初沒想到的東西」未來都可以加進去而不破壞舊資料**。

Hickey 在這件事上很激進——Datomic 的 schema 本身就是 datom,意思是「schema 也是會變的事實」。你不一定要走那麼遠,但精神可以借:**substrate 的硬骨架越少越好,只硬要求真正絕對不能變的東西**。

對你的真正絕對不能變的東西可能只有:

- Event 必須有時間戳(沒有時間就不是 event)
- Event 必須是 append-only(沒有 immutability 就沒有 substrate)
- Event 必須有 source identifier(不然 provenance 斷掉)
- Event 必須有唯一 id(不然沒法 reference)

剩下的——payload 結構、actor schema、causality 表示、attribute 命名規則——都可以是 soft 的,可以後改的。

## 五、第四個關鍵 pattern:故意留 escape hatch

長壽的 substrate 都會留幾個「我知道我會後悔但現在沒時間想清楚」的後門:

**Escape hatch 1:任意 metadata 欄位**

每個 event 帶一個 `metadata: map<string, any>` 欄位,沒有 schema。任何當下沒想清楚但可能有用的東西先丟進去。未來如果發現某個 metadata 欄位很常用,再 promote 成 first-class field。

Postgres 有 jsonb 欄位、MongoDB 整個就是這個哲學、所有長壽的 wire protocol(HTTP、SMTP)都有 extension header 機制。這不是 hack,是有意識的演化策略。

**Escape hatch 2:Reserved namespace**

定義一些 reserved 的 attribute prefix(例如 `_internal_*`、`_experimental_*`),這些東西的語義允許隨時改,使用者跟第三方知道不能依賴它們。讓你有實驗的空間。

**Escape hatch 3:Capability negotiation**

substrate 跟它的 client(lens app、capture source、AI agent)之間有個版本協商機制——client 開頭問「你支援什麼 capability」,substrate 回答一個 list。新 capability 加進來時,舊 client 看不到但不會壞。

HTTP/2、TLS、PostgreSQL wire protocol 都這樣做。對你來說可能 overkill,但記住這個 pattern 存在,後面如果發現需要就拿出來用。

## 六、第五個關鍵 pattern:Ship 早 + 保留改動權的具體做法

把上面這些東西打包成一個具體策略,你可以這樣做第一版:

**Substrate v0.1 的最小完備清單:**

1. **Event format**:
    
    ```
    {
      "id": <uuid>,
      "schema_version": "0.1",
      "time": <iso8601 with timezone>,
      "source": <string, free-form>,
      "actor": <string or structured ref>,
      "type": <string>,
      "payload": <open map>,
      "metadata": <open map>
    }
    ```
    
    只硬要 id、schema_version、time。其他全部允許 null 或 empty。
    
2. **Storage**:sqlite 一張 events 表,加上幾個 index(時間、source、type)。檔案位置標準化在使用者本機某處。
    
3. **API**:
    
    - `append(event) -> id`
    - `read(filter, time_range) -> stream<event>`
    - `replay(from_time, to_time) -> stream<event>`
    - 完全沒有 update、delete、aggregate
    - 所有 derive 是 client 自己做的,substrate 不管
4. **明確 spec 寫死的事**:
    
    - Event immutable
    - id 全域唯一
    - schema_version 必填
    - time 必須帶時區
5. **明確 spec 寫「未定義/未來決定」的事**:
    
    - Source identifier 的命名規則
    - Actor 的 schema
    - Causality 怎麼表達
    - Retraction / forgetting 怎麼處理
    - 多裝置 sync
    - 加密 boundary
    - Derive 的標準介面
    - Provenance tracking 的標準

寫成 spec 的時候,**對「未定義」的東西要明確標記 `deferred to v0.2+, do not depend on stability of this area`**。這句話是你的法律保障——使用者如果在這些區域建東西,他們知道自己是在實驗區。

這個 v0.1 大概兩到三個月可以做出來(包括 spec、reference impl、minimal CLI)。它能做的事很少(基本上就是「append event、query event」),但它能做的事是穩的,不會被未來改動 break。

**然後 v0.2、v0.3 慢慢補上面那些 deferred 的東西**,每次補的時候 raw event log 都還在,使用者的歷史不會丟。新功能 layer 在 raw event 之上,不替換 raw event。

---

## 七、給你最關鍵的一句話

如果上面太多,記住一件事就好:

**Ship 一個故意極簡但永遠不會背叛使用者的 raw event log,把所有複雜決定推遲到之後的 layer。**

「永遠不會背叛使用者」的具體意思是:**這個使用者今天 capture 的 event,五年後不論 substrate 怎麼改,都能被讀出來,而且能用新的 derive 邏輯重算出有意義的東西。**

只要這個承諾守住,你想改什麼都可以。Derive 邏輯第二年整個重寫,沒問題。Lens app 第三年換隱喻,沒問題。Aggregate schema 第四年加一倍,沒問題。**因為使用者真正在乎的東西(他們的 event log)是穩的,其他都是這個 log 之上的詮釋,詮釋換了不影響 log 的價值。**

而且這個策略還有一個 bonus:**它讓 substrate v0.1 的承諾範圍小到可以真的 ship**。你不用第一天就想清楚 derive 邏輯、aggregate schema、UI 隱喻、商業模式——你只要想清楚「什麼是 event,event 怎麼存,event 怎麼讀」這三件事,其他都可以後補。

這把 substrate 路線的「必須完備」壓力從「全部完備」降到「最小核心完備」,而最小核心可能就只有 raw event 那一層。你過去兩年想的所有東西(D0/D1/D2、Guard、sedimentation、aggregate、pattern、trait)**全部都是 raw event 之上的 layer**,可以等 substrate v0.1 ship 出去之後慢慢長,而且每一層都可以是 throwaway 的、可以被後來的 better 版本替換的。

---

**最後一個比較鋒利的 reframe:**

你前面焦慮「substrate 必須完備」,但其實你混淆了兩個東西:

- **Substrate 的承諾範圍必須完備**(我承諾的東西必須做到,不然會背叛使用者)
- **Substrate 的功能範圍必須完備**(我必須一開始就涵蓋所有 use case)

第一個是真的必須,第二個不是。第二個是你自己加給自己的標準,而它是錯的——所有長壽的 substrate 都是從一個小到不能再小的功能範圍開始的(git 第一版只能 commit 跟 checkout,IPFS 第一版只能存單一檔案,Datomic 第一版的 Datalog 只支援基本 join),它們長大是因為承諾範圍守得住,所以可以慢慢加功能而不傷使用者。

你需要的不是「想清楚所有東西再 ship」,是「**想清楚我承諾什麼,然後 ship 那個最小的承諾**」。其他想不清楚的東西不需要 ship、不需要承諾、明確標為 deferred 就好。這個區分一旦清楚,substrate 路線就不再是「兩年想到完備」的可怕長路,而是「兩個月 ship 一個小但堅實的承諾,然後在這個承諾上慢慢長五年」的合理工程。

# Verbs vs Nouns: Why Personal Systems Need a Different Ontology

---

## 起點：對 Littlebird 的不適感

Littlebird.ai 是 2026 年的 ambient capture + AI memory 產品。$11M seed，macOS native，透過 Accessibility API 讀取螢幕上的結構化文字，建一個 text-based working memory，使用者可以問「我今天幹了啥」「哪些 email 重要」。

技術上它做得很好。但作為一個想過個人系統兩年的人，第一反應是：**我不想把資料累積在那裡**。

這個反應不是在嫌功能不夠。是在感覺到一個更深的東西——Littlebird 的 memory 結構上會讓你**越用越虧**。累積越多，context 越厚，memory 越個人化；但同時，你離開的成本越高，而你能帶走的東西不增反減。因為越上層的 derived memory 越專屬於他們的 schema，就算 export 也沒地方用。

每多用一天，你對系統的依賴 +1，你的可攜資產 +0。這是個**負複利**結構。

---

## 兩種 Memory

Littlebird 的問題不在技術，在 ontology——它把 memory 的主詞放錯了。

**Littlebird 模式**：AI 擁有 memory，使用者透過 AI 訪問 memory。Memory 的主詞是 AI——「為了讓這個 assistant 回答得好，它需要知道什麼」。所有 schema、retention、retrieval 都是為了 LLM query 優化的。使用者是 memory 的來源，不是擁有者。

**另一種可能**：使用者擁有 memory，AI 是使用者派去操作 memory 的工人。Memory 的主詞是使用者——「我這個人的沉積物」，AI 只是剛好能讀它的一個 client。

這兩種模式表面看都是「AI + 你的 data」，但權力結構完全相反。

第一種模式下，你不可能「自由用 AI 管理系統」，因為 AI 就是系統，你管理不了系統本身。第二種模式下這是 default 行為，因為 memory 從一開始就是個可被任何 agent 操作的標準化 substrate。

Littlebird 因為走的是 SaaS lock-in 路線，兩件事它都做不了，而且永遠不會做。即使他們意識到這是個缺陷，他們的商業模式不允許。

---

## Obsidian 的核心限制

Obsidian 是另一個極端——local-first、使用者所有權、markdown 檔案可攜。它做到了「儲存所有權」，但做不到另一件事。

Obsidian 最近的 Bases、CLI、MCP 支援看起來很強。有人用 Claude + MCP 一次更新了 1400 個 daily notes——看起來已經是「AI agent 可操作的個人 memory 系統」了。但仔細看之後，有個根本問題浮出來。

**Obsidian 的基本單位是「檔案」，不是「事件」。**

整個 Obsidian 的世界觀建在「note 是個 stable 的東西，使用者打開、編輯、儲存」這個假設上。檔案是名詞，是物件，是有名字、有 path、有 mtime 的實體。所有功能——backlink、graph view、Bases、search——都在這個「名詞宇宙」裡運作。

但人的生活不是檔案，是事件流。你今天看了什麼、跟誰講了什麼、想了什麼、停在哪裡多久，這些是 stream，不是 documents。它們有時間戳、有 actor、有 sequence、有 causal chain。**它們的本質是動詞不是名詞。**

要把事件流塞進「檔案」這個模型，你只能做兩件事：

1. **把事件聚合成檔案**（例如 daily note）——但你失去了事件的時間粒度跟結構，所有東西被壓平成一頁文字
2. **每個事件都做一個檔案**——幾百萬個檔案，Obsidian 的 index、search、graph 全部會崩，而且 markdown link 不適合表達它們的關係

**這就是為什麼 Obsidian 永遠做不出 ambient capture。** 不是他們不想，是 primitives 跟 capture 想表達的東西在 ontological level 不對盤。

Bases、CLI、MCP 都是在「檔案宇宙」裡加工具，它們不會改變底層的 ontology。Obsidian 之所以是 Obsidian，正是因為它選了檔案這個 unit。這個選擇不能撤回，撤回就失去現在所有 5M 使用者愛它的東西。

---

## 名詞宇宙 vs 動詞宇宙

Obsidian 碰到的限制不是個案，是一個在很多領域都反覆出現的根本對立。

### 一般形式

世界可以被描述為兩種東西：

- **名詞（Nouns）**：存在的物，things that _are_
- **動詞（Verbs）**：發生的事，things that _happen_

這兩種描述理論上等價——任何動詞流都可以聚合成狀態，任何狀態都可以分解成歷史。但實作上不等價：你選哪個當基本單位，會決定整個系統能做什麼跟做不到什麼，而且一旦選了就很難換。

### 在電腦科學裡的化身

**程式語言：**

- Noun-first：OOP。世界是一堆 objects，objects 有 state，你呼叫 method 改變 state。基本問題：「這是什麼？它有什麼？」
- Verb-first：Functional / FRP。世界是一堆 transformations，state 是 transformations 的結果。基本問題：「發生了什麼？然後呢？」

**資料庫：**

- Noun-first：RDBMS。基本單位是 row，代表 entity 的當前狀態。歷史是附加品。
- Verb-first：Event sourcing / append-only log。基本單位是 event，當前狀態是把所有 event replay 出來的衍生物。歷史就是本體。

**作業系統：**

- Noun-first：Unix file system。世界是 files、directories、inodes。
- Verb-first：systemd journal、syscall trace。世界是 messages、events、signals 在 process 之間流動。

這幾組對立有個共通 pattern：在每一個領域，noun-first 都「先贏」——因為它對人類直覺友好，容易視覺化，工具鏈成熟早。然後 verb-first 在十年後重新出現，因為某些 noun-first 解不掉的問題逼出了它。Event sourcing 是 RDBMS 用了三十年才開始流行的；FRP 是 OOP 主導二十年後才有人認真做的。

Verb-first 通常解的問題是：

- 「為什麼現在的狀態長這樣？」（audit、debug、time travel）
- 「同樣的歷史能不能 derive 出不同的視圖？」（read model 多態）
- 「事情發生的順序本身是有意義的嗎？」（temporal logic、causality）

而這三個問題，**剛好就是「自我」這個東西要回答的問題。**

### 在哲學裡的化身

**亞里斯多德傳統（noun-first）**：世界由 substance（實體）組成，實體有 attributes，變化是 attributes 的改變但 substance 維持同一性。「這顆蘋果」是一個 stable thing，它從綠變紅是它的 property 變了，但它還是同一顆蘋果。

**赫拉克利特 / 過程哲學（verb-first）**：「人不能踏進同一條河兩次。」基本單位不是物，是流變（becoming）。「河」這個名詞是個方便的抽象，實際上每一刻的河都是不同的水分子組成的不同事件流。Whitehead 的 process philosophy 把這個推到極致：真實的不是 things，是 events；things 是 events 的 derivative。

**佛教（extreme verb-first）**：「無我」就是這個立場推到極致——根本沒有「自我」這個 substance，只有五蘊不斷生滅的過程。你以為的「我」是這些過程聚合起來看起來有連續性的錯覺。

---

## 為什麼「自我」特別反抗名詞宇宙

對大部分東西，你選名詞或動詞都行。一張椅子當名詞處理沒問題，一場演講當動詞處理也沒問題。但**自我這個東西特別頑固地反抗 noun-first 處理**，有三個原因：

**1. 自我沒有穩定的 substrate。**

你的細胞每七年換一輪，記憶會被改寫，價值觀會變，十歲跟三十歲共用的「東西」非常少。如果你硬把自我當 noun，你會發現這個 noun 沒有可以掛 attribute 的 stable core——所有 attribute 都在變，連 ID 都不靠譜。

**2. 自我的同一性來自過程而不是物質。**

你之所以是「你」，不是因為你有某個不變的本質，是因為你的經驗有 continuity——昨天的你記得前天、今天的你延續昨天、這個 chain 不斷續著。**Continuity 是 verb，不是 noun。** 當 chain 中斷（失憶、夢、麻醉），自我就消失了，即使物質還在。

**3. 自我的內容是 derivative，不是 primitive。**

你以為你「有」一個 personality、「有」一些 traits、「有」一些 preferences，但這些東西都是從你過去的行為跟反應裡 derive 出來的。「我喜歡咖啡」不是個原始事實，是「過去 N 次面對咖啡時我有 P 比例選擇喝它」的 summary。**所有 trait 都是 event stream 的 read model**，不是反過來。

這三點加起來：**任何想要服務「自我」這個東西的系統，如果用 noun-first 架構，都會在 ontology 層遇到漏水。**

---

## Obsidian 漏在哪

Obsidian 是個 noun-first 系統，它把人的思考當成一堆「可命名、可定位、可長期存在的物件」來管理。對 PKM 來說這沒問題，因為知識本來就比較像名詞——一個概念、一個定義、一篇參考資料，這些是 stable 的東西。

但對「自我」來說不行，因為自我不是知識。自我是：

**時間流**：你今天某個瞬間想到的事情，跟昨天某個瞬間想到的事情，中間的順序跟間隔本身有意義。Noun-first 系統會把這個時間結構壓平。

**未被命名的東西**：你生活中 99% 的東西不會被你停下來命名。一個還沒成形的不舒服、一個重複出現的小猶豫、一個無意識的回避模式——這些在被命名之前就應該被捕捉，但 noun-first 系統的入口是「建立一個 note 並命名」，**沒被命名的東西根本進不來**。

**派生的 pattern**：「你是個會在壓力下沉默的人」這種觀察不是某個 note 的內容，是橫跨幾百個 event 之後浮現的 pattern。Noun-first 系統可以儲存這個觀察（寫成 note），但無法生成這個觀察（因為原始 event 從來沒進來過）。

這三個漏點不是 bug，是 noun-first ontology 的必然後果。**你不能在一個只接受名詞的系統裡儲存動詞。**

---

## Verb-First 的個人系統長什麼樣

如果一個個人系統真的選 verb-first，那它的 primitives 不是檔案，是這些東西：

**Event（基本單位）**：一個有時間戳的、不可變的、原子的觀察。

- `2026-04-07T14:23:12Z, source=slack, action=read, target=msg_id_xyz, dwell=12s`
- `2026-04-07T14:25:01Z, source=screen, app=Cursor, file="adiabatic/d0.rs", scroll_distance=240px`
- Event 永遠不刪、不改，只 append。這是 event sourcing 的核心。

**Stream（events 的有序集合）**：按 source、按 actor、按 thread 分組。

**Aggregate（從 stream derive 出來的當前狀態）**：

- 「過去七天我跟誰互動最多」
- 「這個月我重訪了哪些主題」
- 同一個 stream 可以 derive 出無限多個 aggregate，每個 aggregate 是一種「看法」

**Pattern（跨 aggregate 的 higher-order 觀察）**：

- 「你週一早上的決策品質明顯比週五低」
- 「你提到 X 的時候情緒會變」

**Trait（極穩定的 pattern）**：

- 「你是個會在壓力下沉默的人」
- Trait 是 derivative 的 derivative，離 primitive 最遠，但對自我認知最重要

在這個 model 裡，**沒有「note」這個基本單位**。你寫下來的東西本身也是一個 event（`source=self, action=write, content=...`），它跟「你看了某個 Slack 訊息」在 ontology 上是平等的，只是 source 不同。

## 方向是不對稱的

有人會問：「為什麼不能兩個都做？Event log 在底下，名詞在上面，兩全其美？」

理論上可以。但實務上方向是不對稱的：**verb-first 可以涵蓋 noun-first，但 noun-first 不能涵蓋 verb-first。**

如果你從 verb-first 開始，加一個 noun layer 是相對容易的——aggregate 自然會 materialize 成可以被命名、被引用、被回看的東西。Datomic、Kafka 上面都可以長 read model，read model 看起來就像 noun-first 系統。

反過來如果你想在 Obsidian 上面加一個 event layer，你會發現 Obsidian 的所有工具（search、graph、Bases、plugin API）都是針對檔案設計的，event 會是個外人。你的 event 沒辦法被 Obsidian 原生功能消費，一旦量大 indexer 會崩。

所以 verb-first 是唯一能兼顧兩種需求的方向。反過來走是死路。

---

## 這對具體架構意味著什麼

### D0 / D1 / D2 的真正意義

沉澱三層不是「資料越往下越精煉」，是「derivation 越往下越穩定」：

- **D0** = raw event stream（verb layer，primary，append-only）
- **D1** = 經過 Guard 篩選 / 標註的 event（promotion = 被認證為值得進入 derive pipeline 的 event）
- **D2** = 從 D0/D1 持續 derive 出來的 aggregate / pattern / trait——這層才開始有「東西」的感覺，但這些「東西」永遠保留著對 underlying event 的 reference

這是 event sourcing 的標準架構（event store → projection → read model），套上了 sedimentation 的隱喻。

### Guard 的角色

Guard 不是「寫入守門員」也不是「沉降守門員」，是 **promotion classifier**：它決定哪些 event 值得進入 derive pipeline。沒被 promote 的 event 不會消失（它們還在 D0，append-only），但它們不會影響 D2 的 derivation。底層 event 永遠都在，你可以隨時改變 Guard 策略然後重新 derive。

### 「Memory 屬於使用者」的技術含義

不是「使用者可以匯出檔案」，是「**使用者擁有自己的 event log，而且有權對這個 log 進行任意 re-derivation**」。其他系統（包括 Obsidian）只能看到 derive 出來的結果，看不到也動不了底層 event。使用者拿到的是 source of truth，不是 view。

---

## 一句話

**Obsidian 在問「我知道什麼」，verb-first personal system 在問「我是誰」。**

前者是 noun question，後者是 verb question。前者可以用檔案系統解，後者必須用 event stream 解。

這不是技術偏好，是 ontology 對齊的問題。而 ontology 不對齊的系統做出來的東西，會在使用者最需要它的時候漏掉最重要的部分——不是因為功能不夠，是因為它從一開始就沒有概念去接住那個東西。

## 為什麼以前沒有人做這個

Verb-based personal system 沒有早出現，是因為它需要五個前置條件同時成熟，而這五個條件直到 2023-2025 才剛好全部到位。

### 前置條件 1：儲存便宜到可以 append 一切

Verb-first 的核心承諾是「raw event 永遠不丟」。但這個承諾只有在儲存幾乎免費的時候才合理。

- 1990 年代：1 GB ≈ $1000，append-only 是經濟自殺
- 2020 年代：1 GB ≈ $0.01，個人五年的 ambient capture（假設一天 100 MB raw）大概 180 GB，一台普通筆電的零頭

Gordon Bell 的 MyLifeBits 計畫從 2001 年開始做，但他需要一台特殊配置的工作站才能存得下。那時候不是想法不對，是物理層面就跑不起來。

### 前置條件 2：Ambient capture 變成可能

Verb-first 需要 capture 是低摩擦的。但這需要技術 stack 配套：

- Audio capture：macOS 跑 Whisper 即時轉錄是 2023 後才實用
- Screen capture（語意層）：Accessibility API 穩定到可以拿來 capture 結構化文字是最近才成熟。Littlebird 證明這個 stack 終於可以信賴
- Mobile / Wearable：Oura、Apple Watch 把生理 signal 變成 stream 是過去五年的事

2010 年想做 verb-first 個人系統，capture source 全部不順手——要嘛要裝奇怪硬體（SenseCam），要嘛要使用者主動操作（失去 ambient 的意義）。Capture 路徑沒有，verb-first 連第一步都跨不出去。

### 前置條件 3：從 raw event 抽出語義的能力（最大的瓶頸）

Verb-first 系統的價值不在 capture，在 derivation。一堆 raw event 本身是垃圾，要能 derive 出有意義的 aggregate / pattern / trait 才有用。

這在 LLM 出現之前是個解不了的問題：

- 規則式系統：只能處理結構化資料
- 統計式 NLP：可以做 sentiment、topic modeling，但每個任務都要單獨訓 model，結果脆弱
- LLM（2022 後）：可以從一段文字裡讀出意圖、情緒、人物、主題、隱含的 pattern，zero-shot

LLM 是 derivation 的 game changer。在它之前，你 capture 一千個 Slack 訊息，只能 keyword search。在它之後，你可以問「我這個月跟 X 的對話有沒有什麼 pattern」。

LLM 真正穩定到個人系統可以依賴它做 production derivation，大概是 2023 後半到 2024。

### 前置條件 4：使用者願意接受「軟體一直在背景看我」

技術成熟還不夠，使用者心理也要成熟。

- 2000 年代：你跟一般人講「裝個軟體一直記錄你做什麼」會被當變態
- 2015-2020：Fitness tracker 普及，量化自己的接受度提升（但僅限健康）
- 2024-2026：Rewind/Limitless/Recall/Littlebird 把「ambient capture」推到主流意識

到 2026 年，「我願意讓某個東西觀察我的數位生活，只要我擁有資料」已經是個可以被討論的選項。十年前完全不可討論。

### 前置條件 5：Local-first 跟使用者所有權的文化基礎建設

Verb-first 個人系統如果走 cloud-only 路線，會掉進 Mem 的陷阱。它真正的形狀需要 local-first 文化的支持。

- 2020：Ink & Switch 發表 "Local-First Software" essay
- 2021-2023：CRDT 從學術走向實用（Automerge、Yjs）
- 2022-2024：Obsidian 從小眾變主流，5M+ 使用者習慣了「我的 vault 是我的檔案」
- 2023-2025：Bluesky、Nostr 等把 user-owned identity 推進主流意識

到 2026 年，有一群足夠大的人已經內化了使用者所有權這個世界觀，不用從零教育。

### 五條曲線剛好全部穿過水平線

這五個條件任一缺一，verb-first 個人系統都跑不起來：

- 沒有便宜儲存 → append-only 經濟自殺
- 沒有 ambient capture → 退化成手動筆記
- 沒有 LLM 可做 derivation → 只能 keyword search，event 流變垃圾
- 沒有使用者心理接受 → 沒有人願意安裝
- 沒有 local-first 文化 → 只能做雲端，然後變 Mem

而這五個條件成熟的時機極度集中在 2023-2025。之前的任何時間點都至少有一條沒過線，所以之前的任何嘗試都會死在那條沒過線的條件上。

---

## 之前的嘗試，他們死在哪裡

**Gordon Bell 的 MyLifeBits（2001-2010）** 直接前身，目標就是 capture 一個人的 lifetime。死在 capture 太重（要穿 SenseCam）、derivation 太弱（2010 年的技術處理不了 lifetime data）、儲存還沒便宜。Bell 自己出了一本書，結論是「想法對但時機不對」。

**Evernote 的 "second brain" 野心（2008-2015）** 想做「remember everything」。選了 noun-first 架構（note 是基本單位），所以 ambient capture 進不來；做了 cloud lock-in，所以 user 流失；沒有 derivation 能力，所以「remember」變成「search」。

**Rewind（2022-2024）** 比較像 verb-first，而且很接近。死在 capture 用 screenshot（太重、太 invasive）、derivation 用早期 GPT（還不夠好）、cloud-first。後來被 Limitless 取代，Limitless 被 Meta 收購。

**Mem（2020-2025）** 最接近 verb-first 的精神。死在拿了 SF 錢被迫走 SaaS lock-in、選了 note 而不是 event 當基本單位（noun-first）、「self-organizing」沒有 LLM 配套。兩年 alpha + final release 之後，變成「更漂亮的 Apple Notes」。

**Roam Research（2019-2022）** 有時間性 daily note 的味道，但還是 noun-first——每個 block 是 named entity。死在 cloud-first、執行問題。

這些失敗有個共通 pattern：每一個都死在五個前置條件中的某一條。Bell 早了二十年。Rewind 早了兩年（LLM 不夠強）。Mem 早了三年（local-first 文化沒成熟）。

---

## 現在的窗口

**沒有任何一個現存玩家同時走 verb-first + local-first + ambient capture + LLM derivation 這四個方向。** 即使每個方向的單獨可行性都已經被別人證明了：

- Littlebird 用了 capture + 文化接受，但選了 cloud + noun-first
- Obsidian 用了 local-first + 使用者所有權，但選了 noun-first
- Mem 用了 LLM + 一點 capture，但選了 cloud + noun-first
- Datomic / XTDB / EventStore 用了 verb-first，但是 enterprise 取向，沒人想過個人應用

這個交集是空的，而且空有理由：走過任一條的人都在各自的社群裡，沒有人剛好同時泡在 local-first、event sourcing、LLM agent、PKM 深度使用者這四個社群。

這個窗口已經開了大約兩年（從 2024 LLM 成熟算起），可能還有一到三年才會被別人填上。之前的等待是有理由的——其他四條曲線還沒爬上來。剩下的等待開始有真實的代價。

---

## 尾聲：兩個不同的問題

市場上現在有兩種「AI + 你的 data」系統，它們表面上像，實際上問的是兩個完全不同的問題：

- **Retrieval over documents**：「幫我從我的筆記裡找到某個東西」。Obsidian + MCP + Claude 可以拼出來。
- **Inference over a life**：「告訴我關於我自己的東西，我自己都沒注意到的」。這需要一個從 D0 開始長起來的系統，因為「a life」不是 documents 的集合，是 events + traits + patterns + relations 的衍生物，而這些東西沒有 capture 跟 sedimentation 就根本不存在。

前者是 noun question，後者是 verb question。前者已經有解了，後者還沒有。

而後者的解，只能從 verb-first 的 ontology 長出來。

## 真實 lineage

**"Malleable software"** — Philip Tchernavskij 2019 PhD thesis 命名，**Geoffrey Litt** 2023 年起在 Ink & Switch 推這個 thesis（PhD MIT CSAIL，現在在 Notion）。Litt 寫過：

> _"the endgame for AI-driven personal software isn't more app silos, it's small tools operating on a shared pool of context about our lives."_
> 
> _"Malleability isn't something one application can easily solve; it requires a new fundamental way that we think about operating systems."_

**這就是你的 thesis 字面版本**。2023 年公開講的。

Project lineage：

- **Wildcard** — 用 spreadsheet 層 customize 任何 web app
- **Potluck** — documents as personal software
- **Embark** — malleable travel planning
- **Riffle** — local-first relational architecture for adaptable apps
- **Stevens** (2025/04) — **這個你必須看**

## Stevens 跟 Adiabatic 撞型

Litt 一年前 ship 的個人 demo Stevens：

- **單一 SQLite table of "memories"，每筆有 optional date** → 你的 event log 簡化版
- **Cron jobs as importers**（Google Calendar / weather / USPS OCR / Telegram / email）→ 你的 ingestion pattern
- **Telegram 當 UI，LLM 組 morning brief** → on-demand view derivation
- **Admin view 是 vibe-coded pixel art game UI** → 字面用 vibe coding 做 personal admin

HN 441 points，ship 在 Val.town，code 公開可 fork。Litt 寫：「Stevens isn't a product you can run out of the box, it's just a personal project I made for myself」 — 但這句話本身就是 signal：**他知道這是 pattern，但他選擇不把它做成 product**。

## 結構性原因（你問的那個）

Litt 那批人沒做成產品，行業也沒做成產品，**確實有 structural 阻力**：

1. **Substrate-application chicken-and-egg**。Malleable apps 要 work 必須有 shared substrate；shared substrate 要值得做必須有 apps 用。冷啟動兩端互相需要對方先存在。Stevens 解法是「我自己一個用戶，自己解 chicken-and-egg」 — 但這就 ship 不出去。
    
2. **Verification gap for non-programmers**。LLM 生成的 code 改你的個人資料，user 怎麼驗證它沒搞壞 data？這個問題在 programmer 用戶身上不存在 (我自己看 diff)，在非 programmer 身上**完全沒解**。所以 malleable software 的 TAM 自動限縮到 programmers 自己。
    
3. **Schema evolution 是 LLM 弱項**。新增 feature 要改 schema，舊 data 怎麼遷移？LLM 寫一次性 code 沒問題，寫長期演化的 migration logic 不可靠。Stevens 用「single table，arbitrary text in memory column」逃避這個問題 — 但這是逃，不是解。
    
4. **Investment 流向錯了**。Cursor / Copilot / Claude Code / Lovable / v0 — 全部 billions 砸在 "make pro devs / non-coders ship apps faster"。**End-user programming for personal use 這條 line 拿到的 funding 接近零**，因為 TAM 故事不性感。Litt 只能在 Ink & Switch (research lab) / Notion (employee) 做這件事，做不了 startup。
    
5. **Distribution model 模糊**。Hosted SaaS? 那 personal data 不在用戶手上。Self-hosted? 99% 用戶不會。Template / library? 那不是 product，是 enablement。**沒有清楚的 "buy this" frame**。
    
6. **No killer use case 證明**。Stevens (家庭日報)、Embark (旅行計畫)、Wildcard (網頁 customize) 都是 nice-to-have。**沒有任何一個 use case 強到讓 user 願意先建 substrate 才能用**。這是最致命的一個。
    
7. **The "good enough for me" trap**。Builders 自己用 malleable software 體驗很好。Ship 給 non-builder 那一瞬間，friction wall 巨大。Stevens 的設計就停在「good enough for Litt's family」。

## Stevens 詳細拆解

[blog post + Val.town 上的 source code 都公開]

**架構**：

- Hosting：Val.town (SQLite + cron + HTTP + email)
- Storage：**單一 SQLite table** 叫 `memories`
    - Columns: text (任意 markdown), tags, creation metadata, **optional date** (when this memory becomes relevant)
    - 沒 date 的 entry 是 background context，永遠 included
- Importers (cron jobs)：
    - 每小時 pull Google Calendar
    - 每小時 weather forecast
    - USPS Informed Delivery email forward → Claude OCR → 寫入 memory
    - Inbound Telegram message → memory
    - Inbound email → memory
    - 每週「fun facts」自動加入
- Output：每天早上組 morning brief 透過 Telegram 推給 Litt 跟他太太
- Brief 組成：撈相關 memories (按 date proximity + always-include) → 餵 LLM → format
- 額外：Pixel-art admin dashboard (vibe-coded with Cursor + Claude 3.7) 顯示 Stevens 在「跑來跑去」處理 memories

**架構特徵 vs Adiabatic OS**：

|維度|Stevens|Adiabatic OS|
|---|---|---|
|Substrate type|Single SQLite table，append-mostly|Event log，verb-first，sedimentation D0/D1/D2|
|Schema|1 table, optional fields, free-text body|Structured ontology + derivation contracts|
|Write path|Direct insert，no validation|Guard single-write path|
|View|LLM compose 每次 from raw memories|Materialized views with derivation logic|
|Use case|Family briefing assistant|General personal substrate|
|Scope|N=1 (Litt 自家)|設計就 N=1 + multi-tenant ready|
|Status|Personal demo, code public, **explicitly not a product**|In dev|

**Litt 自己怎麼定位 Stevens**：

- _"Stevens isn't a product you can run out of the box, it's just a personal project I made for myself"_
- _"小工具操作於 a shared pool of context about our lives — that's the endgame"_
- 用 Stevens 證明：simple architecture (1 table + cron jobs) 已經能跑

**Stevens 沒解的部分** (=你的空間)：

- **Schema evolution**：用 free-text body 逃避結構化，所以新 feature 不需要 migrate。代價：搜尋 / 結構化查詢困難
- **Integrity**：直接 insert，沒有 invariants
- **Multi-source conflict resolution**：Calendar、weather、user input 都直接寫進同一個 table，沒有 reconciliation 層
- **Production-grade durability / privacy**：跑在 Val.town，trust Val.town infra
- **Long-term sedimentation**：append-only memories 沒有 D0→D1→D2 的層次化處理。一年後資料量怎麼辦未解
- **多用戶**：完全沒設計
